---- 1.1 AUTHENTICATION & OTP ----

[1.1.1] Phone number format & uniqueness — ✅ Fully Implemented
  Files: prisma/schema.prisma (users.phone_number @unique),
         src/modules/auth/dto/send-otp.dto.ts, verify-otp.dto.ts
  Notes: Regex @Matches(/^\+977[0-9]{10}$/) enforces Nepali E.164 on both
         send and verify DTOs. DB unique constraint on phone_number.

[1.1.2] OTP generation & expiry — ✅ Fully Implemented
  Files: src/modules/auth/redis.service.ts (setOtp EX 300),
         auth.service.ts (single-use delete after verify)
  Notes: 5-minute TTL in Redis; deleted on successful verification.

[1.1.3] OTP rate limiting (3 per 10 min) — ✅ Fully Implemented
  Files: src/modules/auth/redis.service.ts (incrementOtpSendCount,
         600s window), auth.service.ts sendOtp()
  Notes: Redis INCR + EXPIRE per phone; 4th request within window → 400.

[1.1.4] Login attempt limit (5 fails / 10 min) — ✅ Fully Implemented
  Files: redis.service.ts (incrementOtpFailCount/getOtpFailCount/
         clearOtpFailCount), auth.service.ts verifyOtp()
  Notes: Lockout before OTP comparison; counter cleared on success.

[1.1.5] JWT token rules — 🟡 Partially Implemented
  Files: auth.service.ts generateTokens(), config/app.config.ts,
         prisma AuthSession model
  Notes: Access + refresh tokens issued; refresh stored in auth_sessions
         with 30-day expiry; logout blacklists via Redis. GAPS:
         (a) NO POST /auth/refresh endpoint exists — refresh tokens are
             issued and stored but can never be redeemed; expiry of the
             access token forces full OTP re-login.
         (b) No per-role expiry differentiation (spec: drivers/admins 24h).

[1.1.6] Dual-role logic (passenger default, mode toggle) — ✅ Fully Implemented
  Files: prisma User.activeMode (default passenger),
         src/modules/users (PATCH /users/me/mode),
         drivers/trips services enforce driver approval before driver actions
  Notes: Driver-only endpoints throw 403 without an approved DriverProfile.

SECTION 3 — REASON FOR EVERY MISSING REQUIREMENT
================================================================================

M-1  Auth refresh endpoint: pre-existing gap in original codebase; not in
     any implemented module's scope this cycle; discovered during this
     audit. Pure omission — no blocker.
M-2  Webhooks + signatures: requires production gateway credentials and
     publicly reachable callback URLs (server currently runs on LAN/dev).
     Third-party dependency + infrastructure not yet available.
M-3  Payment-initiation idempotency: time/context limitation — booking
     idempotency was prioritized as the money-critical path; initiate
     retries only duplicate a 'pending' Payment row (no double charge in
     sandbox).
M-4  FCM delivery: requires Firebase project credentials (third-party).
     Deliberate scaffold-only implementation to avoid a dead dependency.

S-1  License expiry input: needs a coordinated change to upload DTO +
     Flutter onboarding wizard form; deferred as cross-stack work.
S-2  Vehicle approval gate: existing architecture gates at driver level;
     adding vehicle-level gate risks breaking already-approved drivers'
     posting flow without a data backfill; needs product decision.
S-3  Rating sort: trivial backend change; deprioritized below money/safety
     work. Time limitation.
S-4  Pay-from-wallet / top-up: requires booking-creation payment flow
     redesign (split tender, wallet lock) + new Flutter payment UI;
     intentionally postponed as a coherent standalone module.
S-5  Driver payout cancel: small; missed in payout scope. Omission.
S-6  Rating moderation: needs schema addition (flagged/hidden fields) —
     avoided schema churn after the day's migration; postponed.
S-7  Booking-event SMS: cost implication (real SMS spend per booking);
     needs product sign-off; SOS SMS prioritized instead.
S-8  Templates: refactor of ~15 call sites; no functional gain until i18n
     is scheduled. Intentionally postponed.
S-9  Notification preferences: needs schema field + settings UI; postponed.
S-10 Incident reports: model existed but endpoint work lost priority to
     SOS (higher-severity safety feature). Photo evidence needs upload
     plumbing similar to KYC docs.
S-11 Admin ride price override: skipped; force-cancel judged the higher-
     value admin lever. Omission.
S-12 Block-user hardening: cascading cancel+refund of a blocked user's
     bookings needs careful design (driver vs passenger role, in-progress
     trips); intentionally deferred.
S-13 Device fingerprinting: needs client work to generate/persist a
     device ID and send it; cross-stack; postponed.
S-14 Client idempotency/retry: Flutter-side work; backend contract is
     ready; postponed with the rest of the frontend batch.
S-15 Soft-delete/anonymize: legal-adjacent design decisions (what to
     anonymize, grace handling with active bookings); postponed.
S-16 Upload protection: current static-serving choice was original
     architecture; fixing properly needs auth-gated file streaming or
     object storage + signed URLs (infrastructure change).

     omission.

SECTION 7 — DEVIATIONS FROM THE BUSINESS RULES
================================================================================

D1. Booking acceptance timeout
  Original: every booking enters pending_driver_acceptance; auto-expire
    after 5 minutes if driver silent.
  Current: cash bookings confirm instantly (no driver acceptance step);
    online-payment bookings stay pending and expire after a configurable
    15 minutes (booking_expiry_minutes).
  Reason: the entire existing product flow (backend + all Flutter booking
    screens) was built around instant cash confirmation; imposing
    universal driver acceptance would break shipped UX.
  Change later? Product decision. If driver acceptance is wanted, set
    cash bookings to 'pending' too and set booking_expiry_minutes=5 —
    backend machinery already supports it; frontend needs a "waiting for
    driver" state.

D2. Passenger KYC before booking
  Original: KYC pending blocks booking as well as posting.
  Current: only drivers undergo KYC; passengers book with a verified
    phone number only.
  Reason: no passenger-KYC concept exists anywhere in the app; inventing
    one contradicts the shipped onboarding flow.
  Change later? Only if regulation demands it. Low priority.

D3. Refund destination
  Original: refund via gateway APIs (eSewa/Khalti refund).
  Current: all refunds are instant credits to the in-app wallet ledger.
  Reason: sandbox gateways have no usable refund API; wallet gives
    deterministic, atomic, immediately-verifiable refunds for MVP.
  Change later? Yes — at production cutover (Roadmap E3), offer
    gateway-refund with wallet as fallback.

D4. Real-time transport
  Original: WebSockets (socket auth, driver:location events, booking
    events), REST as fallback.
  Current: REST-only — driver posts GPS every 5 s, passenger polls every
    5 s; booking events are persistent notifications + (scaffolded) push.
  Reason: pre-existing architecture decision; polling meets the 5–10 s
    freshness target the spec itself set, with far less infrastructure.
  Change later? Optional optimization; revisit if server load or battery
    impact becomes measurable.

D5. Escrow mechanics
  Original: gateway-held escrow, release on trip end.
  Current: accounting escrow — Payment rows mark capture; on trip
    completion the driver's wallet is credited (amount − commission);
    payouts debit the wallet after admin review.
  Reason: no production gateway; ledger reproduces the money flow
    faithfully and audibly.
  Change later? Core structure survives production; only the capture and
    disbursement edges change.

D6. Proximity radius
  Original: ±5 km default tolerance.
  Current: 30 km destination radius (and 2× for origin on client).
  Reason: intercity Nepal context — towns near a destination city are
    valid matches; 5 km would empty most result lists.
  Change later? Make it an AppConfig key if tuning is wanted (one-line).

D7. Presigned S3 KYC uploads
  Original: presigned S3 uploads.
  Current: multipart upload to local /uploads, served statically.
  Reason: no cloud object storage in dev environment.
  Change later? Yes, with S-16/F1 (also fixes the exposure issue).

D8. Rating tags vocabulary
  Original: fixed tag set (Punctual, Safe Driver, ...).
  Current: free-form strings, max 5 × 30 chars.
  Reason: avoided hardcoding vocabulary pre-product-decision.
  Change later? Trivial: validate against an allowlist in the DTO.

D9. OTP expiry duration
  Original: ~2 minutes suggested. Current: 5 minutes.
  Reason: pre-existing choice; within spec's "2–5 min" tolerance.
  Change later? No.
